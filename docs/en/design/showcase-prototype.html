What I learn

My backend learning path, through one real build

This is a backend course. I write TypeScript and Node services end to end — and meet every architecture idea in plain words when my build needs it, not as theory up front.

  • Coding
    Backend development in TypeScript & Node

    I build real services: HTTP APIs, message handlers, data models, and the glue between them — with strict types, a shared workspace, and code I am comfortable showing in an interview.

  • Software design
    Domain-driven design, in plain words

    DDD sounds academic; in practice it means deciding what my system is about before coding it: naming the concepts, drawing boundaries between responsibilities, and keeping each rule in exactly one place. I practice it on a domain small enough to hold in my head.

  • Architecture
    Hexagonal architecture — swap anything, break nothing

    My core logic shouldn't care which database, message broker, or AI provider it talks to. I learn ports and adapters by actually swapping implementations — in this system, even the AI agent is just an adapter.

  • APIs
    Contract-first APIs and events

    I write the OpenAPI and AsyncAPI contracts before any service exists, so any language could implement them — then keep my code honest against them. This is how teams with more than one service stay sane.

  • Distributed systems
    Messaging that survives the real world

    Real systems deliver the same message twice, crash halfway, and retry at the worst moment. I learn idempotency, correlation, and dead-letter queues — the topics senior backend interviews actually probe.

  • Testing
    Tests that define behavior, not decorate it

    Sixteen deterministic acceptance scenarios pin down times, IDs, and every message that must — and must not — be published. My implementation is done when they pass, not when it looks done.

  • Working with AI
    Using AI agents without trusting them blindly

    The theme that ties it together: agents propose, independent verification checks, humans decide. I learn to build that boundary in code — a skill that gets more valuable as agents write more software.

Is this for me? I need comfortable JavaScript or TypeScript basics. Everything else — DDD, architecture, messaging, testing discipline — is taught from zero through the project. It's a backend course: no frontend framework required.

What I build

PatchQuest — the project that carries the course

PatchQuest is the vehicle, not the destination. It's a small platform where a person assigns a coding task to an AI agent and stays in charge of the result. One loop, four hand-offs — and each hand-off teaches me one part of backend engineering.

  1. I write the taskHuman

    What to build, which files may change, how many tries the agent gets, and which checks must pass. Locked once opened — changing it means a new version. Teaches: modeling rules as code.

  2. An agent takes it onAgent

    It borrows the task for a limited time and reports progress. If it goes silent, the task can be handed to another attempt. Teaches: leases, timeouts, and ownership.

  3. It hands in one exact patchAgent

    The change is fingerprinted, so it can't be swapped later — and handing it in twice does nothing the second time. Teaches: immutability and idempotency.

  4. A separate verifier grades itVerifier

    Every required check runs against that exact patch. The agent that wrote it is never allowed to grade it. Teaches: independent verification and evidence.

  5. The system recommendsSystem

    Approve, or ask for changes — advice backed by the evidence, never a decision. Teaches: separating advice from authority.

  6. I make the callHuman

    Nothing ships without a person saying yes to that exact patch and that exact evidence. Teaches: human-in-the-loop by construction.

    The hard parts

    I also build what happens when it goes wrong

    Anyone can build the happy path. The course treats failures as first-class lessons — each one is its own acceptance scenario I make pass.

    • The agent touches forbidden files

      The scope check fails deterministically — the agent can't waive the rule.

    • The agent goes silent

      Its lease expires; late work is rejected, and a retry needs remaining budget.

    • The same patch arrives twice

      The recorded result returns — nothing happens twice.

    • A required check fails

      A failed verdict with evidence. No approval is possible from that run.

    • I reject passing work

      My call outranks the checks. Evidence is kept; a retry carries my feedback.

    • A message can't be processed

      It parks in a dead-letter queue and is replayed only through an audited operation.

    • The verifier's machine dies

      "Aborted" is neither pass nor fail — and can never turn into a recommendation.
